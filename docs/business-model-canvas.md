# Business Model Canvas

## Project Title
SmartTimetable — A Real-Time Class Schedule &amp; Reporting Platform for RIC Students and Lecturers

## 1. Customer Segments
- **Students** who need a reliable, always-current view of their weekly class schedule.
- **Lecturers/staff** who need to report or publish changes to their own class sessions (room, time, cancellation).
- **Academic/admin staff** who manage the master timetable, review student-submitted reports, and resolve scheduling conflicts across departments.

## 2. Value Proposition
SmartTimetable replaces the static, easily-outdated PDF timetable and scattered chat-group announcements with a single, searchable, always-up-to-date schedule. It solves three problems at once: students no longer show up to the wrong room or a cancelled class; lecturers/admin have one clear channel to publish changes instead of several; and every class carries a live status label (Confirmed / Room Changed / Cancelled) so users know at a glance whether they can trust it. The crowd-sourced reporting flow also means the system self-corrects faster than waiting for official admin updates alone.

## 3. Channels
- Mobile-friendly web app (no install required), accessed via a shared campus link/QR code.
- Promoted through student orientation, lecturer briefings, and class representatives.
- Linked from existing campus channels (student portal, class group chats) during rollout so it augments rather than competes with current habits.

## 4. Customer Relationships
- Self-service for students: browse, search, and report issues without needing to contact anyone directly.
- Lightweight trust-building through transparency — every status change shows what changed and when, so users learn to rely on the system over time.
- Admin/lecturer side is relationship-driven: admins review and act on student reports, closing the loop by updating the record so the same issue isn't reported repeatedly.

## 5. Revenue Streams / Sustainability Logic
SmartTimetable is not a paid product; its value is operational and institutional rather than monetary:
- **Operational value:** reduces time lost to students arriving at the wrong room/time and reduces repetitive questions to the security/admin office.
- **Institutional value:** gives academic admin a single, auditable source of truth for schedule changes instead of informal channels (chat groups, verbal announcements) that leave no record.
- **Efficiency value:** crowd-sourced reporting surfaces schedule problems faster than waiting for manual admin checks, lowering the admin team's workload over time.
- Long-term sustainability would come from the institution absorbing hosting/maintenance as part of normal IT services, similar to how it already maintains the student portal.

## 6. Key Resources
- A small development team (this project group) to build and maintain the web app and prototype.
- Basic web hosting and a lightweight database for class records and reports.
- Access to accurate baseline timetable data from the academic office to seed the system.
- Design assets/UI kit already built for the prototype (HTML/CSS wireframes, status badge system).

## 7. Key Activities
- Building and maintaining the schedule display, search, and status-labeling features.
- Building and moderating the student report submission flow.
- Building the admin dashboard for reviewing reports and updating class records.
- Keeping timetable data synced with the academic office's official schedule.
- Testing usability across student and admin user flows (per the wireframe usability checklist).

## 8. Key Partners
- **Academic/Registrar office** — source of official timetable data and approval to publish schedule changes.
- **IT/campus services** — for hosting, domain access, and long-term maintenance if adopted beyond the prototype.
- **Lecturers** — as a source of first-hand cancellation/room-change information.
- **Student representatives** — to help promote adoption and surface early usability feedback.

## 9. Cost Structure
- Development time (the team's own hours) for building and testing the prototype — the main cost at this stage.
- Web hosting and database costs if deployed beyond a local/demo prototype.
- Ongoing maintenance time to keep timetable data accurate each term.
- No licensing costs anticipated at MVP stage since the stack (HTML/CSS/basic backend) uses free/open tools.

## Final Note
Every part of this business model is scoped to what the MVP wireframes already demonstrate: the Home screen delivers the value proposition directly to students; the Schedule and Detail screens deliver the core "always up-to-date, searchable schedule" value; the Report flow is the crowd-sourced mechanism that keeps the system accurate without heavy admin overhead; and the Admin Dashboard/Edit screens are where the institutional partner (academic office) captures and closes the loop on reported issues. Nothing in this canvas assumes features beyond what the prototype already shows — the business model is a description of the same MVP, not a separate roadmap.
